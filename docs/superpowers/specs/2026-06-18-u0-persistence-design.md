# U0 核心数据持久化设计

## 1. 目标与范围

本设计将现有 U0 内存状态升级为可迁移、可重启保留的关系型数据库持久化层，同时保持现有 REST API、统一错误响应、多租户和授权行为不变。

本轮只覆盖 U0 核心数据：

- `tenants`
- `users`
- `roles`
- `user_tenant_roles`
- `tenant_feature_entitlements`
- `tenant_data_grants`
- `audit_logs`
- `idempotency_records`

U1-U7 业务表、生产身份协议、生产对象存储和外部系统连接不在本轮范围内。

## 2. 技术方案

采用 SQLAlchemy 2 作为 ORM，Alembic 管理迁移。正式配置面向 PostgreSQL/PostGIS；自动化测试使用 SQLite 临时数据库，以便快速、隔离地执行契约测试。

数据库 URL 由环境变量提供。代码不写死生产地址或凭证。SQLite 仅作为开发和测试后端，不作为生产数据库验收依据。

## 3. 模块边界

- `apps/api/app/database.py`：创建数据库引擎、会话工厂和请求级会话。
- `apps/api/app/models.py`：定义 U0 SQLAlchemy 表模型、约束和索引。
- `apps/api/app/repositories.py`：封装租户、授权、审计和幂等读写，API 不直接拼装 SQL。
- `apps/api/app/bootstrap.py`：幂等初始化平台租户、客户租户、演示用户、角色和默认功能授权。
- `apps/api/app/main.py`：通过依赖注入使用仓储层，保留当前路由和错误契约。
- `alembic/` 与 `alembic.ini`：提供数据库版本迁移入口。

现有 `src/drone_inspection/foundation.py` 继续承载领域规则和现有验收测试。本轮不进行无关重构；API 的持久化实现通过仓储层复用相同的固定功能码和领域错误码。

## 4. 数据模型与约束

### 4.1 租户、用户与角色

`tenants` 使用字符串主键，保存名称、`tenant_type` 和状态。`tenant_type` 只允许 `PLATFORM_OPERATOR`、`CUSTOMER_TENANT`。

`users` 保存稳定用户标识、显示名称、主租户和状态。认证协议仍为待确认项，本轮只保留演示 Token 到用户标识的映射，不把演示认证声明为生产认证。

`roles` 保存稳定角色编码和名称。初始化角色按源文档角色建议登记为 `PLATFORM_ADMIN`、`PLATFORM_OPS`、`ALGORITHM_ENGINEER`、`ANNOTATION_ADMIN`、`FLIGHT_ADMIN`、`FLIGHT_OPERATOR`、`ANALYSIS_VIEWER`、`AUDIT_VIEWER`，后续不得仅凭租户类型推导权限。

`user_tenant_roles` 通过用户、租户和角色三列唯一约束防止重复授权，并使每次角色查询显式带租户范围。同一用户可在不同租户拥有不同角色。平台方跨租户操作必须同时满足平台方租户身份、允许跨租户操作的角色、受控 `X-Tenant-Id` 和审计四项条件。

本轮权限矩阵固定为：功能授权和数据授权管理接口仅 `PLATFORM_ADMIN` 可写；平台租户的 `PLATFORM_ADMIN`、`PLATFORM_OPS` 可受控切换租户；审计管理查询允许平台租户的 `PLATFORM_ADMIN`、`AUDIT_VIEWER` 使用。客户审计查看员仅查询本租户审计的接口属于后续 U0 增量，不在本轮新增路由。

### 4.2 功能授权与数据授权

`tenant_feature_entitlements` 对 `tenant_id + feature_code` 设置唯一约束。`feature_code` 只接受项目固定枚举；平台租户初始化全部功能，客户租户初始化三个默认功能。

`tenant_data_grants` 保存 `id`、授权方租户、被授权方租户、`data_scope`、用途集合、`effective_from`、`expires_at`、授权状态及审计时间。`data_scope` 至少支持 `asset_types` 和 `sample_ids`，使用 JSON 列保存。授权判定必须同时匹配租户、当前时间位于有效期、有效状态、用途、资产类型和样本子集。

源文档尚未登记数据授权状态枚举。本轮沿用现有 `ACTIVE`，不自行新增撤销状态或撤销接口；新增状态值和撤销流程必须先形成签字变更记录。到达 `expires_at` 后授权自动失效并返回 `DATA_GRANT_412`。

### 4.3 审计与幂等

`audit_logs` 显式包含 `tenant_id`、操作者、动作、资源类型、资源标识、客户端 IP、`request_id`、错误码、状态变化字段和操作时间。拒绝操作也必须写审计；`request_id` 与 API 响应及应用日志保持一致。审计日志保留策略不得少于 180 天，本轮只实现可查询的持久化记录，不实现自动清理。

`idempotency_records` 保存 `tenant_id`、HTTP 方法、稳定路由标识、幂等键、请求指纹、JSON 响应和创建时间，并对 `tenant_id + HTTP方法 + 路由标识 + Idempotency-Key` 设置联合唯一约束。相同作用域、相同键和相同指纹返回原响应；相同作用域、相同键和不同指纹返回 `IDEMP_409`。不同租户或不同写接口可安全复用相同键。

首次成功业务写入、成功审计和幂等响应记录位于同一事务。并发相同请求依赖联合唯一约束串行化结果；唯一约束竞争后重新读取既有记录，并按指纹执行重放或冲突判断。

## 5. 请求与事务流程

每个请求获取独立数据库会话。认证解析用户标识后，仓储层加载用户的归属租户和当前租户角色；受控 `X-Tenant-Id` 切换只允许具备对应跨租户权限的平台方账号。

读请求在仓储层统一附加租户条件。写请求先验证操作者、目标租户、角色、功能或数据授权和幂等键，再执行业务写入与审计。成功时统一提交；普通业务领域错误回滚业务事务。

拒绝、越权和跨租户切换失败的审计不能与被回滚的业务事务共用提交边界。请求层先回滚业务事务，再使用独立短事务写入拒绝审计；只有拒绝审计提交完成后才返回对应领域错误。若审计写入失败，记录同一 `request_id` 的安全日志并保持原拒绝结果，绝不因审计故障放行请求。

数据库唯一约束冲突由仓储层转换为现有固定错误码，不向接口泄露 SQL、表名、连接信息或其他租户资源是否存在。

## 6. 初始化与迁移

Alembic 首个迁移创建八类 U0 表、外键、唯一约束和必要索引，并提供可执行的 `downgrade`。应用初始化数据使用幂等 upsert 语义，可重复执行而不生成重复租户、角色或授权。

快速测试通过临时 SQLite 文件验证关闭并重新创建应用后数据仍保留。迁移测试还必须在开发 PostgreSQL 容器执行 `upgrade -> downgrade -> upgrade`，验证 JSON、外键、唯一约束和事务行为。PostgreSQL 冒烟测试只证明迁移兼容性，不代表政务云生产数据库已经验收。

## 7. 错误与安全边界

- API 错误继续使用 `code/message/request_id/details`。
- 未认证返回 `AUTH_401`，无平台管理权限返回 `PERM_403`。
- 租户越权或不存在继续返回 `TENANT_403`/`TENANT_404`，且不泄露其他租户数据。
- 未授权功能和数据分别返回 `FEATURE_403`、`DATA_GRANT_412`。
- 幂等冲突返回 `IDEMP_409`。
- 数据库内部异常不得返回原始堆栈或 SQL 文本。

数据库连接在启动或健康检查阶段不可用时，服务必须 fail-fast 并保持未就绪，不对外提供业务流量。运行期未知数据库故障需要统一 5xx 错误码，但当前固定错误码表没有通用内部错误码；该错误码必须通过项目方签字变更后登记。本轮不得复用含义不符的现有错误码，也不得自行新增错误码。正常领域错误和已登记错误码流程不受该待确认项影响。

## 8. 测试与验收

实施采用测试先行，至少覆盖：

1. 迁移能够在空数据库创建全部 U0 表及约束。
2. 初始化可重复执行且不会重复写入。
3. 应用重启后功能授权、数据授权、审计和幂等响应仍保留。
4. 客户租户只能读取本租户上下文，不能通过请求切换租户。
5. 平台方受控切换租户并写入审计。
6. 重复授权由唯一约束或幂等规则安全处理。
7. 相同幂等键的不同请求返回 `IDEMP_409`。
8. 不同租户或不同写接口可以复用相同幂等键，并发相同请求只产生一次业务写入。
9. 授权未生效、已过期、用途不符、资产类型不符或样本越界均返回 `DATA_GRANT_412`。
10. 拒绝请求回滚业务数据但保留包含 `request_id`、客户端 IP 和操作时间的审计记录。
11. PostgreSQL 容器完成 `upgrade -> downgrade -> upgrade` 迁移冒烟测试。
12. 审计保留配置不低于 180 天，且不存在自动提前删除路径。
13. 现有 46 项自动化测试保持通过。

## 9. 完成条件

首个 Alembic 迁移及回滚、数据库会话、八类模型、仓储层、幂等初始化和 API 接入全部完成；拒绝审计具有独立提交边界；新增 SQLite 持久化测试、PostgreSQL 迁移冒烟测试与现有回归测试通过；文档明确通用数据库 5xx 错误码待确认项及 PostgreSQL 生产待验收边界；代码提交并同步至当前 GitHub 功能分支。
