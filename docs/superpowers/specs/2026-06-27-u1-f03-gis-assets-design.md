# U1-F03 GIS Minimum Asset Registry Design

## Goal

U1-F03 delivers the minimum GIS asset registry required by the U1 flight-control scope. It provides tenant-scoped road, bridge, and slope asset master data, plus coordinate-transform configuration records for later route planning, mission binding, media analysis, and event-to-asset review.

This increment does not implement a real GIS engine, automatic spatial matching, PostGIS geometry operations, customer GIS synchronization, route planning, mission approval, or L2/L3 precise localization. GIS data source, authoritative CRS, update frequency, and stake mapping remain pending confirmation.

## Source Requirements

- U1 must include GIS minimum assets alongside DJI gateway, device registry, routes, missions, approvals, telemetry, exceptions, and media sync.
- Core tables must include `road_assets`, `bridge_assets`, `slope_assets`, and `coordinate_transforms`.
- All business tables must carry or inherit `tenant_id`.
- Customer tenants may only access assets in their own tenant.
- Platform operators may switch tenant only through controlled `X-Tenant-Id`, role authorization, and audit.
- `FLIGHT_CONTROL` gates flight-control domain reads and writes.
- Fixed error codes must be reused. GIS validation failures use `GIS_422`; idempotency conflicts use `IDEMP_409`; unauthorized feature access uses `FEATURE_403`; hidden cross-tenant detail access uses `TENANT_404`.

## Scope

### In Scope

- Persistent SQLAlchemy models and reversible Alembic migration for:
  - `road_assets`
  - `bridge_assets`
  - `slope_assets`
  - `coordinate_transforms`
- Tenant-filtered repository and service module for GIS asset registry behavior.
- Customer and platform read APIs:
  - `GET /api/v1/gis/road-assets`
  - `GET /api/v1/gis/road-assets/{asset_id}`
  - `GET /api/v1/gis/bridge-assets`
  - `GET /api/v1/gis/bridge-assets/{asset_id}`
  - `GET /api/v1/gis/slope-assets`
  - `GET /api/v1/gis/slope-assets/{asset_id}`
  - `GET /api/v1/gis/coordinate-transforms`
  - `GET /api/v1/gis/coordinate-transforms/{transform_id}`
- Platform-admin write APIs:
  - `POST /api/v1/admin/gis/road-assets`
  - `PATCH /api/v1/admin/gis/road-assets/{asset_id}`
  - `POST /api/v1/admin/gis/bridge-assets`
  - `PATCH /api/v1/admin/gis/bridge-assets/{asset_id}`
  - `POST /api/v1/admin/gis/slope-assets`
  - `PATCH /api/v1/admin/gis/slope-assets/{asset_id}`
  - `POST /api/v1/admin/gis/coordinate-transforms`
  - `PATCH /api/v1/admin/gis/coordinate-transforms/{transform_id}`
- Idempotent platform-admin writes using `Idempotency-Key`.
- Audit records for successful creates/updates and denied reads/writes.
- Automated tests covering migration, constraints, tenant filtering, role/feature enforcement, idempotency, validation, and OpenAPI path exposure.

### Out of Scope

- Real PostGIS geometry columns or spatial indexes.
- Automatic coordinate transformation, stake calculation, or nearest-asset matching.
- `asset_segments`, `event_asset_links`, routes, route versions, missions, approvals, telemetry WebSocket, or media sync.
- Customer write permissions.
- External customer GIS REST/database-view integration.
- Batch or file-based offline GeoJSON import. This increment keeps `geom_json`, `source`, `crs`, and `data_version` as future import metadata fields only.
- Any claim that L2/L3 localization or production GIS acceptance is complete.

## Data Model

The project keeps the current lightweight SQLAlchemy pattern. Geometry is stored as `geom_json` JSON so SQLite tests and PostgreSQL migration smoke tests remain portable. The field is a GeoJSON-style placeholder and must not be treated as authoritative PostGIS geometry.

### `road_assets`

- `id`: string primary key, generated by service as `road_<hex>`.
- `tenant_id`: foreign key to `tenants.id`, indexed.
- `route_code`: required route identifier.
- `road_name`: required road name.
- `direction`: required enum-like value: `UP`, `DOWN`, `BIDIRECTIONAL`, `RAMP`, or `UNKNOWN`.
- `start_stake`: required integer meter value.
- `end_stake`: required integer meter value.
- `geom_json`: required JSON object.
- `source`: required text, default `MANUAL_IMPORT`.
- `crs`: required text, default `PENDING_CONFIRMATION`.
- `data_version`: optional text.
- `created_at`, `updated_at`: UTC timestamps.

Constraints:
- `end_stake >= start_stake`.
- `(tenant_id, id)` unique for future composite references.
- `(tenant_id, route_code, direction, start_stake, end_stake)` unique to prevent duplicate road ranges.

### `bridge_assets`

- `id`: string primary key, generated as `bridge_<hex>`.
- `tenant_id`: foreign key to `tenants.id`, indexed.
- `bridge_code`: required tenant-scoped bridge code.
- `bridge_name`: required bridge name.
- `route_code`: required route identifier.
- `stake_no`: required integer meter value.
- `geom_json`: required JSON object.
- `structure_parts`: required JSON list, default empty list.
- `source`, `crs`, `data_version`, `created_at`, `updated_at`.

Constraints:
- `(tenant_id, id)` unique.
- `(tenant_id, bridge_code)` unique.
- `stake_no >= 0`.

### `slope_assets`

- `id`: string primary key, generated as `slope_<hex>`.
- `tenant_id`: foreign key to `tenants.id`, indexed.
- `slope_code`: required tenant-scoped slope code.
- `route_code`: required route identifier.
- `start_stake`: required integer meter value.
- `end_stake`: required integer meter value.
- `side`: required value: `LEFT`, `RIGHT`, `BOTH`, or `UNKNOWN`.
- `geom_json`: required JSON object.
- `source`, `crs`, `data_version`, `created_at`, `updated_at`.

Constraints:
- `(tenant_id, id)` unique.
- `(tenant_id, slope_code)` unique.
- `end_stake >= start_stake`.

CRS values are constrained to `WGS84`, `GCJ02`, `CGCS2000`, and `PENDING_CONFIRMATION`. The registry stores CRS metadata only; it does not execute coordinate conversion.

### `coordinate_transforms`

- `id`: string primary key, generated as `ctr_<hex>`.
- `tenant_id`: required foreign key to `tenants.id`, indexed. Platform-default records are owned by the platform operator tenant instead of using `NULL`, so tenant isolation, audit, and idempotency remain explicit.
- `source_crs`: required text.
- `target_crs`: required text.
- `method`: required value: `CONFIG_ONLY`, `AFFINE`, `HELMERT`, or `MANUAL_REVIEW`.
- `params_json`: required JSON object.
- `version`: required text.
- `status`: required value: `ACTIVE`, `DEPRECATED`, or `PENDING_CONFIRMATION`.
- `created_at`, `updated_at`.

Constraints:
- `(tenant_id, source_crs, target_crs, version)` unique.
- `source_crs != target_crs`.

## Validation

Service-level validation supplements database constraints because error responses must use project error codes:

- Empty strings in required text fields return `MISSION_422`.
- Invalid enum-like values return `MISSION_422`.
- Invalid CRS values return `GIS_422`.
- Invalid stake ranges return `GIS_422`.
- `geom_json` must be a JSON object with a string `type`. Unsupported or missing geometry type returns `GIS_422`.
- Supported placeholder geometry types are `Point`, `LineString`, `Polygon`, `MultiLineString`, and `MultiPolygon`.
- Customer-provided `tenant_id`, `status`, `created_at`, or `updated_at` fields are forbidden by strict Pydantic request models and return unified `MISSION_422`.
- Duplicate tenant-scoped asset codes or ranges return `MISSION_422`.
- Duplicate tenant-scoped coordinate-transform versions return `MISSION_422`.

## Access Control

Read APIs:
- Require authentication.
- Resolve tenant using the existing tenant-context mechanism.
- Require `FLIGHT_CONTROL`.
- Customer tenants can only read their current tenant.
- Platform operators can use controlled `X-Tenant-Id`.
- Detail reads that cannot find an asset in the current service tenant return `TENANT_404` and write denial audit without revealing whether another tenant owns the ID.

Write APIs:
- Require authentication.
- Require `PLATFORM_ADMIN`.
- Resolve target tenant through controlled `X-Tenant-Id`.
- Require target tenant to have `FLIGHT_CONTROL`.
- Require `Idempotency-Key`.
- Use tenant-scoped idempotency fingerprints.
- Write success and denial audits.

## API Payloads

Responses include `tenant_id` for platform traceability, matching the device registry precedent. Request bodies never accept `tenant_id`.

List responses return:

```json
{
  "items": []
}
```

Asset detail responses return the asset fields directly, including `created_at` and `updated_at` ISO strings.

## Testing Strategy

The implementation must follow TDD:

1. Add failing persistence tests before model/migration/service code.
2. Add failing API tests before HTTP routes.
3. Run focused tests at each step and verify expected failures before implementation.
4. Run all tests, compile checks, and `git diff --check`.
5. Run Linux PostgreSQL migration smoke test syntax and, if feasible, cloud PostgreSQL `upgrade/downgrade/upgrade` smoke test after pushing the branch.

Minimum acceptance evidence:
- U1-F03 focused persistence and API tests pass.
- Full project test suite passes.
- Alembic SQLite migration round-trip passes.
- PostgreSQL migration smoke script covers the new revision.
- Test report clearly states that real GIS source, authoritative CRS, customer GIS integration, L2/L3 localization, routes, missions, and production GIS acceptance remain out of scope.

## Risks and Boundaries

- Storing `geom_json` is deliberate for this increment. It preserves asset references and keeps tests portable, but it is not spatial indexing or precise GIS calculation.
- `coordinate_transforms` records configuration intent only. No code may use them to claim actual CRS conversion accuracy until authoritative CRS and method parameters are confirmed.
- Stake values are stored as integer meters. If the customer later requires formatted stake notation or sub-meter precision, a follow-up migration may be needed.
- Routes and missions should reference these assets later, but this increment must not introduce route or mission behavior.
