# 验收优先级与 U0-U7 推进策略

## 1. 结论

本项目正式开发与验收采用：

```text
M1 先交付模拟闭环
M2 再交付真实 14 算法
U0-U7 按门禁推进
长期准备项可提前并行
```

核心原则：

- U0-U7 的交付门禁必须按顺序。
- 人员和资料准备不必机械串行。
- U6 真实算法、冻结验收集、国产卡、DJI 真实接入、外部系统接口可提前并行准备。
- 不得绕过 U0-U7 门禁声明完成。
- 不得把模拟能力写成 M2 或 U7 生产验收通过。

## 2. M1 / M2 / U7 优先级

| 里程碑 | 优先级 | 目标 | 验收口径 |
| --- | --- | --- | --- |
| M1 | 第一优先级 | 用模拟 DJI、模拟算法跑通平台主链路 | 平台闭环可用，允许模拟算法 |
| M2 | 第二优先级 | 接入真实 14 算法 | 必须真实模型、冻结验收集、指标、回放、抽检、双硬件适配 |
| U7 | 最终验收 | 系统级生产验收 | 性能、安全、弱网、租户隔离、功能授权、数据授权、网络边界、UAT |

## 3. 为什么先 M1 再 M2

先交付 M1 的原因：

1. 当前真实 DJI、真实算法、国产卡、外部系统接口都存在待确认项。
2. 如果一开始追 M2，会被模型、验收集、硬件和生产接口阻塞。
3. M1 能先验证平台架构、租户权限、状态机、媒体链路、分析工单、标注训练流程。
4. M1 通过后，真实算法接入才有稳定承载平台。
5. 模拟算法服务可验证链路，但不得计入 M2。

## 4. U0-U7 门禁顺序

### 4.1 必须严格遵守的门禁

| 顺序 | 必须原因 |
| --- | --- |
| U0 → U1 / U2 | 没有租户、权限、审计、幂等，飞控和媒体不能正式落地 |
| U1 / U2 → M1 | 飞控、媒体、分析、工单闭环是 M1 核心 |
| U3 → U5 / U6 | 没有统一推理契约，真实算法和模型发布会失控 |
| U4 → U5 | 没有样本和数据集，训练和评估缺来源 |
| U5 → U6 | 没有模型版本、发布、回滚，真实算法无法验收 |
| U6 → U7 | 14 算法不达标不能进入最终验收 |
| U7 最后 | U7 是系统级验收，必须建立在 U0-U6 完成基础上 |

### 4.2 不允许的行为

不允许：

- U0 未完成就跳过租户和权限做业务接口。
- U1 未完成就声明真实 DJI 飞控可用。
- U2 未完成就声明平台主链路闭环。
- U3 未完成就让不同硬件后端返回不同结构。
- U4 未完成就做不可追溯训练数据集。
- U5 未完成就发布模型。
- U6 未完成就进入 U7 最终验收。
- 使用模拟算法冒充真实模型通过 M2。
- 使用候选硬件池冒充国产卡实测通过。

## 5. 可并行准备项

以下事项可以提前并行，不必等待对应开发单元开始：

| 并行动作 | 可何时开始 | 说明 |
| --- | --- | --- |
| U6 真实算法模型收集 | U0 开始后即可 | 不阻塞 M1，但要提前推进 |
| 冻结验收集采集与标注 | U1 / U2 开始时即可 | U6 长周期风险项 |
| 国产卡候选评估 | U3 契约设计时即可 | 当前显卡清单为候选池，不是最终锁定 |
| DJI 真实接入资料准备 | U1 模拟器开发时即可 | 证书、API、设备 SN、网络策略提前准备 |
| 外部系统接口调研 | U0 开始即可 | OA、GIS、工单、统一身份、通知 |
| U7 性能 / 安全测试方案 | U2 / U3 稳定后即可 | 测试方案可提前，正式执行需系统稳定 |
| 前端演示页面 | U0 API 契约确定后即可 | 支撑客户阶段性展示 |

并行准备项不得替代正式门禁。

## 6. 推荐阶段计划

### 阶段 1：U0 基础工程

目标：

- 工程骨架。
- 登录认证。
- RBAC。
- 租户隔离。
- 功能授权。
- 数据授权。
- 审计。
- 幂等。
- OpenAPI。
- 基础前端框架。

门禁：

- 客户租户访问训练 / 发布返回 `FEATURE_403`。
- 未授权客户数据训练返回 `DATA_GRANT_412`。
- 写操作有审计。
- 基础服务可启动。

### 阶段 2：U1 + U2 主链路

目标：

- DJI 模拟网关。
- 飞控任务状态机。
- 设备 / 航线 / 任务。
- 媒体入库。
- 分片上传。
- 抽帧。
- 模拟分析。
- 事件。
- 工单。

门禁：

- M1 平台闭环可演示。
- 弱网 / 重复回调 / checksum 有测试。
- 业务平台不直接访问 DJI API。

### 阶段 3：U3 + U4

目标：

- gRPC Predict 契约。
- NVIDIA / 国产模拟后端一致性。
- 标注任务。
- 样本池。
- 质检。
- 数据集封版。

门禁：

- `PredictResponse` schema 一致。
- `Detection.geometry` 三选一。
- `FROZEN` 数据集不可修改。
- 客户样本越权拒绝。

### 阶段 4：U5

目标：

- 训练任务。
- 模型评估。
- 模型版本。
- runtime_artifacts。
- 灰度发布。
- 回滚。
- 数据授权校验。

门禁：

- 未达标模型不能发布。
- 未授权客户数据不能训练。
- 发布和回滚有审计。

### 阶段 5：M1 验收

目标：

- 平台模拟闭环演示。
- U0-U5 测试文档。
- 模拟算法回放。
- 演示账号。
- 演示环境部署。

验收输出：

- U0-U5 单元测试文档。
- 平台闭环演示记录。
- DJI 模拟器回放记录。
- 媒体、抽帧、模拟推理、告警、工单、复核记录。
- 标注、数据集、训练、评估、发布、回滚演示记录。

### 阶段 6：U6 一期 14 算法接入

目标：

- 14 个真实算法逐个接入。
- 冻结验收集 checksum。
- 离线指标报告。
- 视频回放。
- 人工抽检。
- 双硬件记录。

门禁：

- 任一算法未达标，U6 不通过。
- 模拟算法不得计入通过。
- 缺冻结验收集 checksum，不得通过。
- 缺回放或人工抽检，不得通过。

### 阶段 7：U7 系统级验收

目标：

- 性能。
- 安全。
- 弱网。
- 租户隔离。
- 功能授权。
- 数据授权。
- 网络边界。
- 双硬件一致性。
- UAT。

门禁：

- 外部生产接口未锁定，不得通过生产验收。
- 硬件未锁定，不得通过生产验收。
- 网络边界未锁定，不得通过生产验收。
- 租户隔离、功能授权、数据授权、网络边界任一项失败，U7 不通过。

## 7. 每单元测试文档要求

每个开发单元必须生成对应测试文档：

| 单元 | 测试文档 |
| --- | --- |
| U0 | 基础工程与权限审计测试文档 |
| U1 | 飞控平台功能测试文档 |
| U2 | 分析平台闭环测试文档 |
| U3 | 推理运行时兼容测试文档 |
| U4 | 标注平台测试文档 |
| U5 | 训练与算法管理测试文档 |
| U6 | 一期 14 算法验收测试文档 |
| U7 | 系统级验收测试文档 |

测试文档必须包含：

- 测试范围。
- 测试环境。
- 测试数据。
- 自动化测试命令。
- 测试结果。
- 不通过项。
- 修改记录。
- 验收结论。
- 待确认项。

## 8. 风险控制

| 风险 | 控制方式 |
| --- | --- |
| 真实算法迟迟不到位 | M1 先用模拟服务，U6 建交付跟踪表 |
| 冻结验收集未确认 | 提前启动采集、标注、checksum 计划 |
| 国产卡未锁定 | 候选池先评估，U3 先做契约，U7 前必须锁定 |
| DJI 接入资料不完整 | U1 先模拟网关，真实接入单独验收 |
| 外部系统接口不确定 | 适配器先行，内置替代兜底，生产需确认 |
| 权限边界被绕过 | U0 作为强门禁，所有业务接口依赖租户与授权 |
| 过早微服务化拖慢联调 | 业务 API 先模块化单体，天然边界服务独立 |

## 10. 正式开发执行原则

后续开发计划、迭代排期、测试文档和验收材料按本文件执行：

1. M1 优先于 M2。
2. U0 是所有业务开发强门禁。
3. U1/U2 组成 M1 主链路核心。
4. U3 是真实算法和双硬件接入基础。
5. U4/U5 支撑数据闭环和模型发布。
6. U6 真实算法不完成，不进入 U7 最终验收。
7. U7 必须最后执行。
8. 资料、模型、硬件、接口可以提前准备，但不得绕过门禁。
9. 每个单元必须生成测试文档并按测试文档验收。
10. 不满足验收项必须修改后重新验收。
