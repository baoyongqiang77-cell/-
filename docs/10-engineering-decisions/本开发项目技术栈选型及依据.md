# 本开发项目技术栈选型及依据

## 1. 结论

本项目建议采用：

```text
React + TypeScript + Vite + Ant Design
FastAPI + Pydantic + SQLAlchemy + Alembic
PostgreSQL + PostGIS
RabbitMQ + Celery
Redis
MinIO / S3 compatible object storage
gRPC + Protobuf
Docker Compose
```

总体架构建议为：

```text
前后端分离 + 模块化单体后端 + 异步任务/推理服务可独立部署
```

该方案优先满足一期正式开发目标：快速形成 M1 主链路闭环、保持多租户/权限/审计/状态机边界清晰，并为 U3 推理运行时、U6 真实算法接入、U7 联调验收预留扩展空间。

## 2. 推荐技术栈

| 层级 | 选型 |
| --- | --- |
| 前端 | React + TypeScript + Vite + Ant Design |
| 后端 API | Python 3.12 + FastAPI + Pydantic |
| 数据库 | PostgreSQL + PostGIS |
| ORM / 迁移 | SQLAlchemy 2 + Alembic |
| 异步任务 | Celery + RabbitMQ |
| 缓存 / 会话 / 实时辅助 | Redis |
| 对象存储 | MinIO 开发测试，生产走 S3 兼容适配层 |
| 内部推理 | gRPC + Protobuf |
| 部署 | Docker Compose 起步，后续可迁移 Kubernetes / 政务云容器平台 |
| 测试 | pytest + 接口测试 + 前端组件测试，后续补 Playwright |

## 3. 选型依据

### 3.1 后端选择 FastAPI

本项目后端需要大量 REST API、统一错误响应、OpenAPI 文档、WebSocket 遥测、异步任务入口和类型化数据校验。FastAPI 基于 Python 类型提示构建 API，并能自动生成 OpenAPI 文档，适合把现有 `docs/00-baseline/API.md` 中的接口基线快速工程化。

同时，项目与视觉算法、推理服务、模型制品、验收脚本关系紧密，Python 技术栈能降低业务后端与算法工程之间的语言割裂。

参考：

- FastAPI 官方文档：https://fastapi.tiangolo.com/
- OpenAPI Specification：https://swagger.io/specification/
- Pydantic 官方文档：https://docs.pydantic.dev/

### 3.2 暂不优先选择 Spring Boot

Spring Boot 在政企项目中成熟度高，但本项目一期核心风险集中在视觉算法、推理运行时、模型制品、双硬件适配和自动化验收。若一开始采用 Java 主后端，会增加算法侧、推理侧、验收脚本侧的协作成本。

因此建议一期正式工程先采用 Python/FastAPI。若客户后续明确要求 Java 技术栈，可将平台管理 API 或网关层逐步迁移或扩展为 Spring Boot 服务。

### 3.3 前端选择 React + TypeScript + Vite + Ant Design

本系统是典型复杂业务后台，页面包含飞控任务、遥测、媒体、分析事件、告警工单、标注、训练、算法、审计、租户权限等模块。React 适合组件化构建复杂 UI；TypeScript 有利于维护接口类型和状态枚举；Vite 提供快速开发和构建体验；Ant Design 提供成熟的企业级后台组件。

参考：

- React 官方文档：https://react.dev/
- Vite 官方文档：https://vite.dev/
- Ant Design 官方文档：https://ant.design/

### 3.4 数据库选择 PostgreSQL + PostGIS

项目数据模型以强关系为主，包括租户、用户、角色、设备、任务、媒体、分析任务、事件、工单、标注、数据集、训练任务、模型版本、发布记录和审计日志。PostgreSQL 适合承载这类事务型业务数据。

同时，系统涉及道路、桥梁、边坡、航线、坐标转换、事件定位、桩号和资产空间映射，因此需要空间数据能力。PostGIS 是 PostgreSQL 的空间扩展，适合支撑 GIS 资产与事件定位。

参考：

- PostgreSQL 官方文档：https://www.postgresql.org/docs/
- PostGIS 官方文档：https://postgis.net/documentation/

### 3.5 ORM 与迁移选择 SQLAlchemy + Alembic

正式项目需要严肃管理核心表、状态枚举、租户字段、索引、审计字段和迁移历史。SQLAlchemy 2 适合 Python 后端的关系建模，Alembic 适合数据库迁移版本管理。

参考：

- SQLAlchemy 官方文档：https://docs.sqlalchemy.org/
- Alembic 官方文档：https://alembic.sqlalchemy.org/

### 3.6 异步任务选择 Celery + RabbitMQ

媒体入库、断点续传、抽帧、分析任务、推理队列、训练任务、模型评估、发布、回滚、通知等都不应长期占用同步 HTTP 请求。RabbitMQ 是成熟消息代理，Celery 是 Python 生态中成熟的分布式任务框架，组合适合一期快速落地异步任务中心。

参考：

- Celery 官方文档：https://docs.celeryq.dev/
- RabbitMQ 官方文档：https://www.rabbitmq.com/docs

### 3.7 Redis 的角色

Redis 建议用于缓存、短期会话、限流辅助、WebSocket 在线状态、轻量进度状态和幂等辅助，不建议替代 PostgreSQL 作为核心业务事实库。核心数据仍应落 PostgreSQL，并通过审计日志和迁移脚本保证可追溯。

参考：

- Redis 官方文档：https://redis.io/docs/

### 3.8 对象存储选择 MinIO / S3 兼容适配层

系统涉及视频、图片、抽帧、数据集导出包、模型制品和评估报告等大文件，不适合直接存储在数据库中。开发测试环境建议使用 MinIO；生产环境通过 S3 兼容适配层对接客户认可的政务云或内网对象存储。

对象路径必须保持租户隔离，例如：

```text
tenant_id/project_id/mission_id/...
```

参考：

- MinIO 官方文档：https://min.io/docs/minio/

### 3.9 内部推理接口坚持 gRPC + Protobuf

项目架构基线已明确内部推理采用 gRPC，且 `PredictRequest` / `PredictResponse` 需要稳定契约，以保证 NVIDIA 后端与国产 AI 后端返回结构一致。gRPC + Protobuf 适合服务间高性能、强契约调用，也便于后续将推理运行时独立部署。

参考：

- gRPC 官方文档：https://grpc.io/docs/
- Protocol Buffers 官方文档：https://protobuf.dev/

### 3.10 部署先用 Docker Compose

一期正式开发应先保证本地和测试环境可一键启动。Docker Compose 可快速组织 API、前端、PostgreSQL/PostGIS、RabbitMQ、Redis、MinIO、DJI 模拟网关和推理模拟后端。

后续进入政务云、私有云或更高并发环境时，再迁移到 Kubernetes 或客户指定容器平台。

参考：

- Docker Compose 官方文档：https://docs.docker.com/compose/

## 4. 架构拆分建议

一期不建议一开始拆成大量微服务。推荐先做模块化单体后端，内部按 U0-U7 拆包：

| 模块 | 职责 |
| --- | --- |
| `foundation` | 租户、认证、RBAC、功能授权、数据授权、审计、幂等 |
| `flight` | 设备、机巢、航线、任务、审批、遥测、DJI 适配网关接口 |
| `media_analysis` | 媒体入库、分片、抽帧、分析任务、事件、工单、复核 |
| `inference` | gRPC 契约、推理任务、后端适配、双硬件一致性门禁 |
| `annotation` | 样本池、标注任务、质检、数据集导出 |
| `training` | 数据集、训练任务、模型版本、评估、发布、回滚 |
| `algorithms` | 一期 14 算法主数据、阈值、验收跟踪 |
| `acceptance` | U0-U7 验收门禁、测试报告、UAT 证据 |

后续可独立拆出的服务：

| 服务 | 拆分时机 |
| --- | --- |
| DJI 适配网关 | 真实 DJI API、证书、回调和 DMZ 网络边界明确后 |
| 推理运行时服务 | NVIDIA / 国产卡测试环境可用后 |
| 训练服务 | 训练资源、模型仓库和队列调度明确后 |
| 媒体处理服务 | 文件量、抽帧量和并发上传明显增长后 |

## 5. 与项目规约的匹配关系

| 项目硬约束 | 技术栈支撑 |
| --- | --- |
| 多租户、功能授权、数据授权 | FastAPI 依赖注入 + PostgreSQL 权限模型 + SQLAlchemy |
| 统一错误响应 | FastAPI exception handler + Pydantic schema |
| 所有写接口幂等 | PostgreSQL 唯一约束 + Redis 辅助缓存 |
| 异步任务中心 | Celery + RabbitMQ + PostgreSQL 任务表 |
| WebSocket 遥测 | FastAPI WebSocket + Redis 在线状态 |
| 媒体和模型制品 | MinIO / S3 兼容对象存储 |
| GIS 资产与事件定位 | PostgreSQL + PostGIS |
| 内部推理统一契约 | gRPC + Protobuf |
| 双硬件适配 | NVIDIA / 国产后端适配器 + runtime_artifacts |
| 自动化测试与验收文档 | pytest + 接口测试 + Markdown 报告 |

## 6. 推荐实施顺序

1. 搭建 monorepo 结构：`backend`、`frontend`、`infra`、`docs`、`tests`。
2. 用 Docker Compose 启动 PostgreSQL/PostGIS、RabbitMQ、Redis、MinIO。
3. 先完成 U0：认证、租户、RBAC、功能授权、数据授权、审计、幂等、OpenAPI。
4. 完成 U1/U2 的 M1 模拟闭环：DJI 模拟网关、任务状态机、媒体入库、抽帧、模拟分析、事件工单。
5. 完成 U3 gRPC 契约和模拟后端。
6. 完成 U4/U5 标注、数据集、训练、发布、回滚门禁。
7. 在真实模型和冻结验收集具备后推进 U6。
8. 在真实硬件、外部接口和网络边界锁定后推进 U7。

## 7. 当前决策

本项目正式开发建议以该技术栈作为一期默认基线。未确认事项继续按项目规约处理：可以先做适配器接口、模拟器、配置占位和契约测试，但不得把模拟能力写成生产已完成能力。
