# 正式代码结构与服务拆分策略

## 1. 结论

本项目正式代码结构建议采用：

```text
前后端分离 + monorepo + 模块化单体起步，少量边界服务独立部署
```

即：

- 不采用传统前后端混合单体。
- 不在一期一开始拆成全量微服务。
- 使用 monorepo 管理前端、后端、适配网关、推理运行时、worker、部署文件、契约和文档。
- 业务 API 后端内部按 U0-U7 模块化组织。
- DJI 网关、推理运行时、异步 worker、scheduler 作为独立部署进程或服务。

## 2. 三个核心问题的决策

| 问题 | 决策 |
| --- | --- |
| 单体应用还是前后端分离？ | 前后端分离 |
| 是否采用 monorepo？ | 采用 monorepo |
| 是否需要多服务拆分？ | 需要拆分天然边界服务，但业务后端先模块化单体 |

## 3. 前后端分离决策

### 3.1 建议

正式工程采用前后端分离：

```text
apps/web  -> React 前端
apps/api  -> FastAPI 后端 API
```

### 3.2 依据

本系统是复杂业务后台，前端页面和交互会持续增长：

- 登录与租户上下文。
- 平台总览。
- 飞控任务。
- 设备与机巢。
- 航线与 GIS 资产。
- 实时遥测。
- 媒体入库与抽帧。
- 分析结果与事件。
- 告警与工单。
- 标注任务。
- 数据集。
- 训练任务。
- 模型版本。
- 算法发布。
- 审计日志。
- 系统配置。

后端同时需要承担：

- REST API。
- WebSocket。
- 租户上下文。
- RBAC。
- 功能授权。
- 数据授权。
- 审计日志。
- 幂等。
- 状态机。
- 异步任务。
- 对象存储。
- gRPC 推理调用。

前后端分离后：

1. 前端可以独立迭代客户演示页面和复杂后台交互。
2. 后端可以专注 API、权限、状态机、审计和任务链路。
3. 前端静态资源可独立部署，适配公有云演示和政务云生产。
4. 后端 API 可独立扩容、测试和审计。
5. 现有 `docs/00-baseline/API.md` 可作为前后端协作契约。

### 3.3 不采用传统混合单体的原因

传统前后端混合单体前期看似快，但本项目后续页面和权限复杂度高，会导致：

- 前端发布与后端发布强绑定。
- 客户演示页面迭代变慢。
- API 契约不清晰。
- 多端复用能力弱。
- 后续迁移到政务云或多环境部署成本更高。

## 4. Monorepo 决策

### 4.1 建议

正式工程采用 monorepo。

推荐顶层结构：

```text
apps/
  web/
  api/
  dji-gateway/
  inference-runtime/
  worker/
  scheduler/
packages/
  contracts/
  shared-config/
infra/
  compose/
  k8s/
docs/
tests/
```

### 4.2 依据

本项目跨模块契约很多，必须保持一致：

- 固定错误码。
- 固定 `feature_code`。
- 固定状态枚举。
- REST API schema。
- gRPC Protobuf。
- 租户与权限模型。
- 审计字段。
- 对象存储路径规则。
- 算法编码。
- U0-U7 验收文档。

采用 monorepo 的好处：

1. 一次变更可同时修改前端、后端、契约、测试和文档。
2. 状态枚举、错误码、API schema、Protobuf 可统一管理。
3. U0-U7 开发单元可以在同一仓库内闭环追踪。
4. 演示环境、正式工程、部署文件和测试报告可统一版本化。
5. 有利于客户审查、交付归档和后续验收追溯。

### 4.3 工具策略

由于本项目包含 Python 后端和 TypeScript 前端，前期建议采用轻量 monorepo：

- 统一目录结构。
- 统一脚本。
- Docker Compose 编排。
- Python 与 Node 各自使用自己的依赖管理。

不建议一开始引入过重 monorepo 工具。等前端包、共享组件、构建任务明显增多后，再评估 Nx 或 Turborepo。

参考：

- Nx 官方文档：https://nx.dev/
- Turborepo 官方文档：https://turborepo.com/docs

## 5. 多服务拆分策略

### 5.1 总体建议

一期不做全量微服务拆分。推荐：

```text
业务 API 后端 = 模块化单体
天然边界能力 = 独立服务或独立进程
```

### 5.2 推荐拆分表

| 服务 / 组件 | 是否独立 | 依据 |
| --- | --- | --- |
| Web 前端 | 独立 | 静态资源独立部署，便于客户演示和版本迭代 |
| API 后端 | 独立服务，内部模块化单体 | 租户、权限、状态机、审计需要统一入口和一致事务 |
| DJI 适配网关 | 独立服务 | 项目规约要求只有 DJI 网关可访问互联网和 DJI Cloud API |
| Inference Runtime | 独立服务 | 内部推理采用 gRPC，NVIDIA/国产后端需要独立资源和契约 |
| Celery Worker | 独立进程 | 抽帧、分析、训练、评估、发布不能阻塞 API |
| Scheduler | 独立进程 | 补偿、重试、状态巡检、清理任务需要独立调度 |
| 标注服务 | 一期不独立 | 先作为 API 模块，避免过早拆分 |
| 分析业务服务 | 一期不独立 | 先作为 API 模块 + worker 任务，负载增长后再拆 |
| 训练管理服务 | 一期不独立 | 管理 API 留在后端，训练执行由 worker 独立处理 |

### 5.3 独立 DJI 网关的依据

项目规约明确：

- 业务平台不得直接调用 DJI Cloud API。
- 所有 DJI 能力必须经“大疆机场 3 适配网关”封装。
- 仅“大疆机场 3 适配网关”允许按白名单访问互联网和 DJI Cloud API。
- 其余业务平台默认运行在政务云内网。

因此 DJI 网关必须具备独立部署边界，后续可放在 DMZ 或受控外联区。

### 5.4 独立推理运行时的依据

项目规约明确：

- 内部推理采用 gRPC。
- NVIDIA 后端与国产后端必须实现统一 `Predict` 契约。
- 国产卡型号未锁定前只能做接口契约和模拟后端验收。
- 同模型双后端输出差异超过阈值时必须阻断发布。

因此推理运行时应独立部署，便于：

- 挂载 GPU 或国产 AI 加速卡资源。
- 独立扩缩容。
- 独立健康检查。
- 独立模型制品加载。
- 与业务 API 保持清晰契约。

### 5.5 模块化单体的依据

租户、权限、功能授权、数据授权、审计、状态机和错误响应必须贯穿所有业务模块。若一期一开始拆成大量微服务，会带来：

- 跨服务事务复杂。
- 租户上下文传播复杂。
- 审计链路复杂。
- 状态机一致性难维护。
- 本地开发和客户演示部署成本升高。
- U0-U5 主链路联调变慢。

因此一期业务 API 后端先采用模块化单体，并用清晰包边界避免后续拆分困难。

FastAPI 支持大型应用使用 `APIRouter` 拆分多文件、多模块，适合该模式。

参考：

- FastAPI Bigger Applications：https://fastapi.tiangolo.com/tutorial/bigger-applications/
- FastAPI APIRouter：https://fastapi.tiangolo.com/reference/apirouter/

## 7. API 后端内部模块

`apps/api/app` 内部建议按以下模块拆分：

| 模块 | 职责 |
| --- | --- |
| `foundation` | 租户、用户、认证、RBAC、功能授权、数据授权、审计、幂等 |
| `flight` | 设备、机巢、航线、任务、审批、遥测、DJI 网关接口 |
| `media_analysis` | 媒体入库、分片、抽帧、分析任务、事件、工单、复核 |
| `annotation` | 样本池、标注任务、质检、数据集导出 |
| `training` | 数据集、训练任务、模型版本、评估、发布、回滚 |
| `algorithms` | 一期 14 算法主数据、阈值、验收跟踪 |
| `audit` | 审计日志查询、导出、保留策略 |
| `common` | 统一错误响应、状态枚举、依赖注入、配置、工具函数 |

## 8. 后续可拆服务时机

| 服务 | 拆分时机 |
| --- | --- |
| 分析业务服务 | 分析任务量明显增大，API 后端成为瓶颈 |
| 标注服务 | 标注工具复杂度上升，需独立前后端协作 |
| 训练管理服务 | 训练任务调度、日志、资源配额变复杂 |
| 审计服务 | 审计数据量大，需要独立归档和查询 |
| 通知服务 | 短信、邮件、企业微信、OA 消息等渠道确定 |

拆分前必须确认：

- 租户上下文传播方式。
- 服务间认证方式。
- 统一错误响应。
- 审计链路。
- 幂等策略。
- 数据一致性边界。
- 回滚策略。

## 9. 与部署策略的关系

本文件与 `docs/10-engineering-decisions/部署环境目标与容器编排选型.md` 配套使用。

公有云演示环境：

- 使用 Docker Compose 编排 `web`、`api`、`dji-gateway`、`inference-runtime`、`worker`、`scheduler` 和依赖服务。
- 重点保证快速部署、快速重置、快速迭代。

政务云生产环境：

- 以 Kubernetes 为目标形态。
- `dji-gateway`、`inference-runtime`、`worker` 等天然边界服务独立部署。
- 业务 API 后端保持模块化单体，后续按负载和边界逐步拆分。

## 10. 最终执行原则

后续正式开发遵循：

1. 前后端分离。
2. 采用 monorepo。
3. 业务 API 后端先模块化单体。
4. DJI 网关独立部署。
5. 推理运行时独立部署。
6. worker 和 scheduler 独立进程。
7. 不在一期过早拆全量微服务。
8. 所有共享枚举、错误码、OpenAPI、Protobuf 和验收文档统一放在 monorepo 内管理。
