# 前端管理后台设计范围与页面规划

## 1. 结论

一期前端正式开发采用：

```text
先做管理后台
先做业务操作型后台
不先做官网式首页
地图视图分阶段实现
Logo 与主题先用稳重默认版，后续按客户素材替换
```

核心原则：

- 前端服务于飞控、媒体、分析、工单、标注、训练、算法、审计等业务操作。
- 首页直接进入系统总览，不做营销落地页。
- 页面设计以清晰、稳定、可追溯、可长期使用为目标。
- 不做只适合汇报的炫技大屏作为主界面。

## 2. 是否先做管理后台

建议：先做管理后台。

依据：

1. 一期目标是 M1 主链路闭环，不是品牌展示官网。
2. 客户演示账号需要展示登录、租户、权限、任务、告警、工单等真实业务界面。
3. 飞控、媒体、分析、标注、训练、算法、审计均属于管理后台场景。
4. 租户隔离、功能授权、数据授权、越权拒绝必须通过后台页面可见。
5. 管理后台能直接承接 U0-U7 开发单元和验收用例。

## 3. 页面优先级

### 3.1 P0 页面

P0 页面用于 M1 演示和主链路闭环，必须优先开发。

| 页面 | 主要内容 | 目标 |
| --- | --- | --- |
| 登录页 | 本地账号、演示账号、租户上下文 | 可登录、可识别角色与租户 |
| 总览页 | 任务、告警、工单、算法交付、测试状态 | 快速看到系统运行与验收状态 |
| 飞控任务页 | 设备、航线、任务状态、模拟 DJI 回调 | 展示飞控主流程 |
| 媒体页 | 媒体入库、抽帧、checksum、状态 | 展示媒体主链路 |
| 分析结果页 | 检测结果、事件、证据图、复核状态 | 展示视觉分析结果 |
| 工单页 | 告警派单、处理、关闭、回流 | 展示告警到处置闭环 |
| 14 算法页 | 算法清单、交付状态、阻断项 | 展示 U6 真实算法交付状态 |

### 3.2 P1 页面

P1 页面用于完善 M1 后半段和 U4/U5/U7 管理能力。

| 页面 | 主要内容 | 目标 |
| --- | --- | --- |
| 标注页 | 标注任务、样本、复检、FROZEN 数据集 | 展示标注闭环 |
| 训练 / 发布页 | 训练任务、模型版本、发布门禁、回滚 | 展示模型管理闭环 |
| 审计页 | 登录、越权、发布、跨租户操作记录 | 展示安全审计 |
| 租户 / 权限页 | 功能授权、数据授权、角色绑定 | 展示平台方授权能力 |

### 3.3 P2 页面

P2 页面用于生产配置和运维增强。

| 页面 | 主要内容 | 目标 |
| --- | --- | --- |
| 系统配置页 | 外部接口、对象存储、通知、运行参数 | 生产配置管理 |
| 外部系统适配页 | OA、GIS、工单、统一身份、通知配置 | 显示接口状态与待确认项 |
| 性能与验收页 | UAT、性能、安全、弱网、网络边界报告 | 支撑 U7 验收 |

## 4. 推荐页面顺序

正式前端开发建议按以下顺序：

```text
登录
→ 总览
→ 飞控任务
→ 媒体
→ 分析结果
→ 工单
→ 14 算法跟踪
→ 标注
→ 训练发布
→ 审计
→ 租户权限
→ 系统配置
```

## 5. 地图视图策略

### 5.1 是否需要地图

需要地图视图，但分阶段实现。

依据：

- 飞控任务需要展示航线和机巢位置。
- GIS 资产需要展示道路、桥梁、边坡。
- 分析事件需要展示资产或定位点。
- 工单需要展示事件位置。
- U7 生产验收需要网络、GIS 和定位证据。

### 5.2 分阶段地图能力

| 阶段 | 地图能力 |
| --- | --- |
| M1 / Demo | 最小地图视图：任务点位、航线示意、资产绑定、事件定位 |
| U1 / U2 | 道路、桥梁、边坡资产图层；任务轨迹；事件点 |
| U7 / 生产 | 接客户权威 GIS、坐标转换、桩号 / 区段定位 |

### 5.3 地图实现原则

1. 地图组件必须可替换。
2. 不在 GIS 来源和坐标系未确认前写死地图供应商。
3. 地图展示坐标与权威资产坐标必须区分。
4. `source_crs`、`transform_status` 和定位等级必须可见。
5. 无法自动定位的事件显示为 `NEEDS_GEO_REVIEW`。

## 6. Logo 与主题策略

| 项目 | 建议 |
| --- | --- |
| Logo | 先使用系统名称 + 简洁文字标识 |
| 正式 Logo | 等客户提供后替换 |
| 主题色 | 深青 / 蓝绿 / 石墨灰 |
| 字体 | 优先系统默认中文字体，保证政务云内网可用 |
| 组件风格 | Ant Design 企业后台风格 |
| 首页风格 | 直接进入系统总览 |
| 图表 / 地图 | 服务业务状态，不做装饰性大屏 |

视觉方向：

```text
稳重
清晰
信息密度适中
业务优先
少装饰
可长期使用
```

## 7. 不建议事项

一期不建议：

- 先做官网式首页。
- 先做纯展示大屏。
- 使用大量动画和装饰性视觉。
- 把地图做成不可替换供应商绑定。
- 在客户 Logo 未确认前硬编码品牌素材。
- 做过度复杂的实时 3D 场景。
- 用营销文案替代业务操作页面。

原因：

- 系统主要用户是平台运维、飞控、分析、标注、算法和审计人员。
- 这些用户需要高效完成任务，而不是观看宣传页。
- 一期验收重点是闭环、权限、状态、审计、算法和网络边界。

| 登录 | 是 | 是 | 是 | 是 | 是 | 是 | 是 |

| 媒体 | 是 | 是 | 可选 | 是 | 是 | 可选 | 只读 |
| 分析结果 | 是 | 是 | 是 | 可选 | 是 | 可选 | 只读 |
| 工单 | 是 | 是 | 否 | 可选 | 是 | 否 | 只读 |
| 14 算法 | 是 | 可选 | 是 | 只读 | 只读 | 只读 | 只读 |

| 训练 / 发布 | 可选 | 否 | 是 | 否 | 否 | 否 | 只读 |
| 审计 | 是 | 是 | 可选 | 否 | 否 | 否 | 是 |

## 9. 页面状态设计

所有业务页面必须体现：

- 加载中。
- 空数据。
- 正常数据。
- 无权限。
- 异常 / 失败。
- 待确认 / 阻断。
- 模拟数据标识。

典型状态：

| 状态 | 用途 |
| --- | --- |
| `PASS` | 测试或规则通过 |
| `BLOCKED` | 缺真实模型、硬件、外部接口或生产材料 |
| `PENDING` | 待提供或待开始 |
| `NEEDS_REVIEW` | 需要人工复核 |
| `NEEDS_GEO_REVIEW` | 需要人工定位复核 |
| `FEATURE_403` | 当前租户未开通目标功能 |
| `DATA_GRANT_412` | 缺数据授权 |

## 10. 与现有本地 Web Demo 的关系

当前 `web-demo` 是演示验证版本，用于：

- 快速展示账号。
- 验证页面顺序。
- 验证角色菜单。
- 验证 M1 模拟闭环展示。
- 验证算法跟踪表和测试报告入口。

正式前端开发应以本文件为基线，在 `apps/web` 中使用 React + TypeScript + Vite + Ant Design 重新实现，不直接把 `web-demo` 当生产前端。

## 11. 与项目规约的关系

本前端规划依据：

1. 一期目标是平台主链路闭环和 14 算法上线验收。
2. 客户租户默认只能使用飞控、标注、分析结果、告警工单。
3. 训练、模型发布、数据集导出、跨租户统计、平台运维默认关闭。
4. 地图与 GIS 数据来源、坐标系、桩号映射仍是待确认项。
5. 演示环境可使用模拟数据，但不得写成生产真实接入。
6. 所有高风险能力必须由后端鉴权和审计兜底。

## 12. 正式开发执行原则

后续前端正式开发按本文件执行：

1. 先做管理后台。
2. 首页直接进入总览。
3. P0 页面优先支撑 M1 闭环和客户演示账号。
4. 地图先做最小可用，保持供应商可替换。
5. 主题使用稳重企业后台风格。
6. 客户 Logo 未提供前使用系统名称文字标识。
7. 前端菜单权限不替代后端鉴权。
8. 模拟数据和待确认项必须在页面中明确标注。
