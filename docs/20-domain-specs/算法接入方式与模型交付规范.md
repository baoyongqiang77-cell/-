# 算法接入方式与模型交付规范

## 1. 结论

一期算法接入采用：

```text
M1 统一模拟算法服务
U3 统一 gRPC 契约
U6 真实模型逐个接入
ONNX 优先作为中间标准
NVIDIA 与国产后端分别生成 runtime artifact
输入、预处理、后处理、输出 schema 全部元数据化
```

核心决策：

| 问题 | 决策 |
| --- | --- |
| 14 个算法是否先接模拟服务 | 是，先接统一模拟算法服务跑通 M1，但不得计入 M2 |
| 真实模型仓库路径 | 建立统一模型仓库或对象存储模型注册路径 |
| 模型格式 | ONNX 作为中间标准，NVIDIA 转 TensorRT，国产卡转厂商格式 |
| 输入尺寸、预处理、后处理 | 每个模型版本必须登记元数据，不得隐含在代码中 |
| 输出 schema | 对齐统一 `PredictResponse`，`geometry` 只能在 bbox / mask / polyline 中三选一 |

## 2. 分阶段接入策略

| 阶段 | 算法方式 | 验收口径 |
| --- | --- | --- |
| M1 | 统一模拟算法服务 | 只用于平台闭环，不计入真实算法验收 |
| U3 | 模拟后端 + gRPC 契约 | 验证 Predict 接口、输出 schema、错误码、超时 |
| U6 / M2 | 真实模型 | 必须通过离线指标、回放、人工抽检、双硬件适配 |
| U7 | 真实模型 + 真实硬件 / 生产环境 | 系统级验收 |

说明：

- 当前平台主链路不能等待 14 个真实模型全部就绪。
- 模拟服务用于验证媒体、抽帧、分析任务、事件、工单、复核、标注回流。
- 模拟算法不得计入 M2 一期最终验收。

## 3. 模拟算法服务

### 3.1 目标

模拟算法服务用于 M1：

- 跑通分析任务。
- 生成标准 `PredictResponse`。
- 生成 bbox / mask / polyline 三类 geometry 示例。
- 验证事件聚合、告警、工单、复核、反馈样本回流。
- 验证错误码、超时、失败和重试。

### 3.2 必须覆盖的场景

| 场景 | 要求 |
| --- | --- |
| 正常检测 | 返回 `SUCCEEDED` 和 detections |
| 空结果 | 返回 `SUCCEEDED`，detections 为空 |
| 推理失败 | 返回 `FAILED` 和统一错误码 |
| 推理超时 | 返回 `TIMEOUT` 和 `INFER_504` |
| 后端未验证 | 返回 `RUNTIME_428` |
| bbox 输出 | 覆盖目标检测类算法 |
| mask 输出 | 覆盖分割类算法 |
| polyline 输出 | 覆盖裂缝、标线等线状目标 |

## 4. 真实模型仓库规范

### 4.1 路径建议

真实模型不建议散落存放。建议统一模型仓库结构：

```text
model-registry/
  phase1/
    traffic_congestion/
      mv_001/
        source/
        onnx/
        nvidia/
        domestic/
        configs/
        reports/
    road_crack/
      mv_001/
        source/
        onnx/
        nvidia/
        domestic/
        configs/
        reports/
```

如模型文件较大，不应直接进入 Git 仓库。Git 中只保留：

- 模型 URI。
- 版本号。
- checksum。
- 元数据。
- 评估报告路径。
- 回放记录路径。
- 人工抽检记录路径。

模型二进制文件存入对象存储、模型仓库或客户认可的制品库。

### 4.2 每个模型版本必须包含

```text
model.onnx
labels.json
preprocess.json
postprocess.json
thresholds.json
runtime_config.json
metrics.json
checksum.sha256
README.md
```

如存在多后端制品，还应包含：

```text
nvidia/model.plan
domestic/model.om
domestic/model.magicmind
domestic/vendor_runtime_package
```

具体国产格式以最终硬件型号和 SDK 为准。

## 5. 模型格式策略

建议采用 ONNX 作为中间标准格式。

| 格式 | 用途 |
| --- | --- |
| PyTorch / 原始格式 | 训练产物或算法团队交付源模型 |
| ONNX | 跨后端中间格式，作为统一转换基线 |
| TensorRT engine / plan | NVIDIA 推理制品 |
| 国产卡格式 | 如 Ascend OM、MagicMind、厂商 GPGPU engine |
| JSON 配置 | 预处理、后处理、阈值、label schema、runtime config |

推荐转换流程：

```text
PyTorch / 原始模型
→ ONNX
→ NVIDIA TensorRT
→ 国产卡厂商格式
→ 双后端一致性测试
```

依据：

- ONNX 是跨框架、跨推理后端的通用交换格式。
- NVIDIA 和多数国产后端具备 ONNX 转换链路或兼容层。
- 有利于统一输入输出、版本管理和一致性测试。

## 6. 输入、预处理、后处理元数据

每个算法、每个模型版本必须登记固定元数据，不得只写在算法代码中。

### 6.1 preprocess.json 建议字段

| 字段 | 示例 |
| --- | --- |
| `input_width` | 640 / 1280 / 1920 |
| `input_height` | 640 / 720 / 1080 |
| `color_format` | RGB / BGR |
| `normalize` | `0-1` 或 mean/std |
| `mean` | `[0.485, 0.456, 0.406]` |
| `std` | `[0.229, 0.224, 0.225]` |
| `resize_mode` | letterbox / stretch / center_crop |
| `keep_ratio` | true / false |
| `pad_value` | 114 |

### 6.2 postprocess.json 建议字段

| 字段 | 示例 |
| --- | --- |
| `postprocess_type` | detection / segmentation / classification / density |
| `confidence_threshold` | 0.70 |
| `nms_iou_threshold` | 0.45 |
| `max_detections` | 300 |
| `mask_threshold` | 0.50 |
| `polyline_simplify_epsilon` | 2.0 |
| `min_area_px` | 16 |
| `review_band` | `[0.50, 0.70]` |

### 6.3 thresholds.json 建议字段

| 字段 | 示例 |
| --- | --- |
| `default_threshold` | 0.70 |
| `per_label_thresholds` | `{ "road_crack": 0.70 }` |
| `min_consecutive_frames` | 2 |
| `review_band` | `[0.50, 0.70]` |
| `tenant_overrides_allowed` | true |

### 6.4 依据

输入、预处理、后处理元数据化的原因：

1. NVIDIA 与国产后端必须保持一致输入输出。
2. 预处理不一致会导致双硬件指标不一致。
3. 后处理不一致会导致事件数量、置信度和几何结构不一致。
4. U6 离线指标、回放和人工抽检需要可追溯。
5. 模型版本回滚后必须复现原始推理行为。

## 7. 统一输出 schema

输出必须对齐项目 gRPC `PredictResponse` 和 `Detection` 结构。

### 7.1 PredictResponse

```json
{
  "inference_job_id": "job_001",
  "status": "SUCCEEDED",
  "error_code": "",
  "backend": "nvidia",
  "model_version_id": "mv_001",
  "latency_ms": 38,
  "detections": []
}
```

字段约束：

| 字段 | 约束 |
| --- | --- |
| `status` | 只能是 `SUCCEEDED`、`FAILED`、`TIMEOUT` |
| `backend` | 只能是 `nvidia`、`domestic` |
| `error_code` | 必须复用统一错误码 |
| `model_version_id` | 必须与请求一致 |
| `backend` | 必须与请求一致，不允许静默切换 |

### 7.2 Detection

```json
{
  "label": "road_crack",
  "confidence": 0.91,
  "geometry": {
    "type": "polyline",
    "points": [[120.1, 88.2], [160.4, 96.7]],
    "image_width": 1920,
    "image_height": 1080
  },
  "attributes": {
    "length_px": "45.8",
    "severity": "medium"
  }
}
```

`geometry` 必须三选一：

| 类型 | 用途 |
| --- | --- |
| `bbox` | 目标检测 |
| `mask` | 分割 |
| `polyline` | 裂缝、标线等线状目标 |

### 7.3 分类 / 密度类算法包装

分类、密度类算法也应包装为 detection-like 结构。

交通拥堵示例：

```json
{
  "label": "traffic_congestion",
  "confidence": 0.86,
  "geometry": {
    "type": "bbox",
    "bbox": [0, 0, 1920, 1080],
    "image_width": 1920,
    "image_height": 1080
  },
  "attributes": {
    "congestion_level": "high",
    "vehicle_density": "0.78"
  }
}
```

## 8. 14 算法接入顺序

不建议 14 个算法同时接入。建议先接 3 个代表类型：

| 类型 | 推荐算法 | 目的 |
| --- | --- | --- |
| 目标检测 | `pedestrian_intrusion` 或 `road_obstacle` | 验证 bbox 输出 |
| 分割 | `road_pothole` 或 `road_water` | 验证 mask 输出 |
| 线状 / 病害 | `road_crack` | 验证 polyline / mask 和复杂后处理 |

打通以下能力后，再批量接剩余 11 个算法：

- 模型注册。
- ONNX 转换。
- NVIDIA runtime artifact。
- 国产后端 runtime artifact。
- Predict 契约。
- 输出 schema。
- 离线评估。
- 回放测试。
- 人工抽检。
- 双硬件一致性。

## 9. 数据库登记要求

每个算法至少登记：

| 表 / 字段 | 要求 |
| --- | --- |
| `algorithms.code` | 固定 14 算法编码 |
| `algorithm_label_schemas` | 标签 schema |
| `algorithm_thresholds` | 默认阈值与租户级覆盖 |
| `model_versions` | 模型版本、指标、checksum、状态 |
| `runtime_artifacts` | NVIDIA / 国产后端制品 |
| `evaluation_report_uri` | 离线评估报告 |
| `replay_record_uri` | 视频回放记录 |
| `manual_sampling_record_uri` | 人工抽检记录 |
| `checksum` | 模型、数据集、报告 checksum |
| `status` | CREATED / VALIDATING / READY / BLOCKED 等 |

`runtime_artifacts.backend` 只能是：

```text
nvidia
domestic
```

## 11. 双硬件一致性

同一模型版本、同一帧输入，NVIDIA 与国产后端输出差异超过阈值时，必须阻断发布。

建议比较项：

| 比较项 | 说明 |
| --- | --- |
| label | 类别一致 |
| detection count | 检测数量差异不超过阈值 |
| confidence | 置信度差异不超过阈值 |
| bbox IoU | bbox 空间重合度 |
| mask IoU | mask 空间重合度 |
| polyline distance | 线状目标距离误差 |
| attributes | 关键属性一致 |

国产卡型号未锁定前：

- 可做模拟后端契约测试。
- 可做 ONNX 转换预检查。
- 不得声明真实国产硬件验收通过。

## 12. 与 U0-U7 的关系

| 单元 | 算法接入重点 |
| --- | --- |
| U2 | 分析任务调用统一推理服务，不绑定具体硬件 |
| U3 | gRPC Predict 契约、NVIDIA/国产后端适配、双硬件一致性 |
| U4 | 标注任务、标签 schema、样本来源 |
| U5 | 数据集、训练、模型版本、发布、回滚 |
| U6 | 14 个真实算法、指标、回放、人工抽检、冻结验收集 |
| U7 | 真实硬件、性能、安全、网络和 UAT |

## 13. 验收要求

### 13.1 M1

可验收：

- 模拟算法服务返回统一输出。
- 分析任务可完成。
- 事件、工单、复核、回流可跑通。

不可声称：

- 真实 14 算法已上线。
- M2 算法验收通过。
- 双硬件真实验收通过。

### 13.2 U6 / M2

每个算法必须具备：

- 真实模型。
- 冻结验收集 checksum。
- 离线指标报告。
- 视频回放记录。
- 人工抽检记录。
- 阈值策略。
- 输出 schema。
- NVIDIA 运行记录。
- 国产后端运行记录或明确阻断说明。

任一一期算法未达标，U6 不通过。

## 14. 依据

本规范依据：

1. 项目规约明确 M1 可使用模拟算法服务。
2. 项目规约明确 M2 必须使用真实模型，模拟算法不得计入通过。
3. 一期 14 个算法编码固定，不得自创编码。
4. U3 规定内部推理采用 gRPC。
5. NVIDIA 与国产后端必须统一 `Predict` 契约。
6. `Detection.geometry` 必须在 `bbox`、`mask`、`polyline` 中三选一。
7. 同模型双后端输出差异超过阈值必须阻断发布。
8. 国产卡最终型号仍未锁定时，只能做模拟后端契约测试。
9. U6 需要离线指标、回放、人工抽检和冻结验收集 checksum。

## 15. 正式开发执行原则

后续 U2、U3、U5、U6 算法相关开发按本文件执行：

1. 先接统一模拟算法服务跑通 M1。
2. 模拟算法不得计入 M2。
3. 真实模型按模型注册表逐个接入。
4. ONNX 作为中间标准格式。
5. NVIDIA 和国产后端分别生成 runtime artifact。
6. 输入尺寸、预处理、后处理、阈值、label schema、output schema 必须元数据化。
7. Predict 输出必须使用统一 schema。
8. bbox / mask / polyline geometry 必须三选一。
9. 先接 3 个代表算法，再批量接入剩余算法。
10. 双硬件一致性未通过不得发布。
